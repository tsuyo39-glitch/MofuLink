# CLAUDE.md — ペット記録アプリ「MofuLink（モフリンク）」

このファイルは Claude Code がこのプロジェクトを実装する際に**最初に読むべきルールと前提**をまとめたものです。
実装の詳細な機能仕様は `SPECIFICATION.md` を参照してください。

---

## 1. プロジェクト概要

自分・家族で使う**個人用のペット記録 PWA（Webアプリ）**。
参考アプリは「ペットノート+」。健康・成長・思い出・お金を一元管理する。

**最重要の方針（既存アプリへの不満を解消することが価値の中心）:**

1. **広告なし・課金なし** — 既存アプリ最大の不満点。完全に無料・買い切り思想で作る。
2. **データを絶対に失わない（機種変更でも消えない）** — アプリ再インストールや iPhone 買い替えでデータが消える事故をなくす。ローカル保存 + 確実なバックアップ/引き継ぎを前提に設計する。これが Phase 2 クラウド同期の主目的。
3. **動作が軽い** — 起動・表示が遅いという不満が多い。初期表示を最優先で軽くする。

---

## 2. 開発・利用環境

- **開発機:** Mac
- **利用端末:** iPhone のみ（ホーム画面に追加して PWA として使う）。PC・タブレットは対象外
- **ターゲットブラウザ:** iOS Safari のみ。これ以外は動作保証しない
- iOS Safari は PWA の挙動に独特の制約があるため（後述）、ここを最優先で作り込む
- iPhone 専用なので、レイアウトは縦画面のスマホ幅に最適化する（PC幅対応は不要）

---

## 3. 技術スタック（確定）

| 領域 | 採用技術 | 備考 |
|---|---|---|
| 言語 | TypeScript | 型でデータ構造の事故を防ぐ |
| フレームワーク | React 18 | 関数コンポーネント + Hooks のみ |
| ビルド | Vite | 開発・本番ビルド両方 |
| スタイル | Tailwind CSS | ユーティリティクラスで統一 |
| PWA化 | vite-plugin-pwa | manifest / Service Worker / オフライン |
| ローカルDB | Dexie.js (IndexedDB) | Phase 1 のデータ保存先 |
| クラウド同期 | Google Drive API (OAuth) | Phase 2 で追加。バックアップ/引き継ぎに使う |
| グラフ | Recharts | 支出・体重の集計表示 |
| 日付処理 | date-fns | 年齢の自動計算など |
| 画像圧縮 | browser-image-compression | 保存前に必ず圧縮（容量対策） |

**重要:** Phase 1 はローカルのみで完結させる。クラウド（Google Drive）は Phase 2 まで導入しない。
ただしデータアクセス層（リポジトリ）を抽象化しておき、保存先をローカル↔Google Drive で差し替えられる構造にすること。

## 4. コーディング規約

- ファイル文字コードは **UTF-8 / 改行コード LF** で統一する（Mac/Windows を行き来するため、混在させない）
- コンポーネントは1ファイル1コンポーネント。`PascalCase.tsx`
- 状態管理はまず React 標準（useState / useReducer / Context）で。ライブラリは必要になるまで足さない
- UI 文言・コメントは**日本語**で書く
- 日付は内部では ISO 文字列（`YYYY-MM-DD`）で保持し、表示時に整形する
- 金額は**整数（円）**で保持する。小数・浮動小数点で金額を扱わない
- 画像は IndexedDB に Blob で保存し、表示時に `URL.createObjectURL` で参照する

## 6. iOS Safari / PWA の注意点（実装時に必ず守る）

- ホーム画面追加時のアイコン・スプラッシュ・`apple-touch-icon` を忘れない
- iOS の IndexedDB は**ストレージ容量に制限があり、長期間使わないと OS にデータを消される**可能性がある
  → だからこそ Phase 2 のクラウドバックアップが「データを失わない」要件の本体になる。Phase 1 では「エクスポート（JSONダウンロード）」機能を必ず付け、手動バックアップ手段を確保する
- iOS の Web Push 通知は対応状況に制約があるため、通知機能は Phase 3 の任意項目とする
- 画像を多用するので、必ず圧縮してから保存する（無圧縮だと容量をすぐ圧迫する）

## 7. 実装の進め方

- `SPECIFICATION.md` のフェーズ順（Phase 1 → 2 → 3）で実装する
- 各 Phase の完了時に「この端末で実際に使える状態」になっていること
- 大きな機能を一度に作らず、Phase 1 の「プロフィール → 日記 → 支出」の順に1機能ずつ動く状態にしてから次へ進む
- 不明点・設計判断が必要な箇所は、勝手に進めず確認すること
